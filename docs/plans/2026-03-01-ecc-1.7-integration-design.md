## 2. TDD Skill Improvements

Merge 6 upstream improvements into `skills/test-driven-development/SKILL.md`. All additive — no existing content removed.

### 2a. Add "Step 0: Define the API" before RED phase

Before writing the first test, define the interface:
- Function signatures with parameter and return types
- Stub with `throw new Error('Not implemented')` (or language equivalent)
- This ensures RED fails for the right reason (not implemented) rather than type errors

### 2b. Add user journey framing (features only)

For new features (not bug fixes), start with:
```
As a [role], I want [action], so that [benefit]
```
This anchors tests to user intent, not implementation details.

### 2c. Document three-tier test taxonomy

| Tier | What | When |
|------|------|------|
| Unit | Functions, modules, pure logic | Always |
| Integration | API routes, DB operations, service interactions | Always |
| E2E | User flows, browser automation | User-facing features (optional, see OPT-OUT.md) |

Follow your project's existing test file organization conventions.

### 2d. Strengthen coverage language

- Change "80%+ good" to "80% minimum standard"
- Add: "100% required for: authentication, payment, security-critical, core business logic"

### 2e. Add wrong/correct test pairs

Three concrete examples:
- Testing implementation details vs user-visible behavior
- Brittle selectors vs semantic selectors
- Test isolation (no cross-test dependencies)

### 2f. Add workflow pipeline note

```
Full workflow: writing-plans → test-driven-development → requesting-code-review → /sp-ecc:test-coverage
```

## 3. Code Reviewer Improvements

Merge 7 improvements across `agents/code-reviewer.md` and the review template.

### 3a. Confidence-based filtering (highest value)

Add at top of agent file:
- Only report if >80% confident it is a real issue
- Skip stylistic preferences unless they violate project conventions
- Skip issues in unchanged code unless CRITICAL security issues
- Consolidate similar issues ("5 functions missing error handling" not 5 separate findings)

### 3b. Add 3 missing security checks

- CSRF vulnerabilities (state-changing endpoints without protection)
- Authentication bypasses (missing auth checks on protected routes)
- Exposed secrets in logs (logging tokens, passwords, PII)

### 3c. Add backend patterns section

When reviewing backend code:
- Unvalidated input (request body/params without schema validation)
- Missing rate limiting on public endpoints
- Unbounded queries (no LIMIT on user-facing endpoints)
- N+1 queries (loop fetching instead of JOIN/batch)
- Missing timeouts on external HTTP calls
- Error message leakage (internal details sent to clients)
- Missing CORS configuration

### 3d. Add review summary table

Every review ends with severity count table + verdict line:
```
| Severity | Count | Status |
|----------|-------|--------|
| Critical | 0     | pass   |
| Important| 2     | warn   |
| Minor    | 1     | note   |

Verdict: WARNING — 2 Important issues should be resolved before merge.
```

### 3e. Add dead code detection

"Commented-out code, unused imports, unreachable branches" under Code Quality.

### 3f. Add BAD/GOOD code examples

Inline examples for top 3 issues: deep nesting (early returns), mutation (immutable), N+1 (JOIN/batch).

### 3g. Model to sonnet

Change `model: opus` to `model: sonnet`.

### Unchanged

Keep: 3-tier severity (Critical/Important/Minor), SHA-based scoping, push-back protocol, architecture checklist, production readiness, workflow integration modes.

## 4. Writing Plans Improvements

Merge 8 improvements into `skills/superpowers/writing-plans/SKILL.md`. All additive.

### 4a. Requirements Restatement as Step 1

Before writing plan content, output: "**Requirements as I understand them:**" block. Human corrects before investing in plan detail.

### 4b. Risks section in plan template

```
**Risks:**
- HIGH: [risk — mitigation]
- MEDIUM: [risk — mitigation]
```

### 4c. Complexity estimate in header

```
**Complexity:** High / Medium / Low
```
No time estimates (unreliable for AI work).

### 4d. Dependency notation in task structure

Optional `Depends on: Task N` field per task. Helps parallelization decisions.

### 4e. Confirmation gate embedded in plan output

Plan document ends with:
```
**READY?** Proceed / Modify: [changes] / Different approach: [alternative]
```

### 4f. Phasing guidance for large features

For large features: MVP (smallest valuable slice) → Core (happy path) → Hardening (edge cases) → Polish (optimization). Each phase should be independently shippable.

### 4g. Testing strategy in plan template

```
**Testing:** Unit: [what], Integration: [what], E2E: [if applicable]
```

### 4h. Plan quality red flags

No testing strategy? No file paths? No dependencies between tasks? Phases that can't ship independently?

### Unchanged

Keep: Bite-sized 2-5 min steps, complete code in plan, exact file paths, machine-readable header, 2-mode execution handoff, DRY/YAGNI/TDD principles.

- Detects duplicates, conflicts, stale entries

## 7. Formatter Hook Upgrade

### 7a. Upgrade `scripts/hooks/prettier-format.js`

Current: runs `npx prettier --write` unconditionally.

New: auto-detect formatter:
1. Check for `biome.json` / `biome.jsonc` → use `npx @biomejs/biome format --write`
2. Check for any of 11 Prettier config variants → use `npx prettier --write`
3. Traverse up from edited file to project root looking for configs
4. No formatter found → skip silently
